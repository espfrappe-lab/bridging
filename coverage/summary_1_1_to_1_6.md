## Description

The generated test suite met 100% of the benchmark tests for all documented behaviors in five of the six subtopics, then went beyond the benchmark with additional EXPANDED tests for every subtopic. In **1.1 Creating Keypairs**, the benchmark directly asserts four behaviors — that `accountId` starts with `G`, that `secretSeed` starts with `S`, that the same seed always produces the same `accountId`, and that a public-key-only keypair's `accountId` matches the input. The generator matched all four, then added 13 EXPANDED tests: 12 covering error handling for invalid seeds and account IDs, the `canSign()` capability flag for both random and seed-derived keypairs, and uniqueness across successive `random()` calls; plus 1 BONUS test that caught a documentation defect — the example seed `SCZANGBA5Y...` has an invalid strkey checksum and the generator confirmed it throws rather than silently accepting it. The benchmark additionally checks that `accountId.length == 56` and `secretSeed.length == 56`, but since neither length requirement appears in `sdk-usage.md`, the generator correctly skipped these two undocumented assertions. In **1.2 Loading an Account**, the benchmark tests eight behaviors covering sequence number presence, balance iteration, native-type detection, amount parsing, and the exists/not-exists error-handling pattern; the generator matched all eight and added eight more EXPANDED tests in total — six explicit ones adding null-checks for `AccountResponse`, an `int.tryParse()` guard on `sequenceNumber`, field-access loop safety for the balance list, and `ErrorResponse` type and status-code assertions; one IMPLICIT test covering the async API contract (all test methods are `async`, inherently verifying `await`-ability); and one PARTIAL test for the non-native `assetCode` branch, which could only confirm the field is accessible rather than asserting a real value because the test fixture lacked a trustline. In **1.4 HD Wallets (SEP-5)**, the benchmark tests six behaviors — the 24-word count of a generated mnemonic, the `G`-prefix of a derived address, uniqueness across different indices, same-index determinism, and that a mnemonic with a passphrase yields a different address than without; the generator matched all six and added seven EXPANDED tests — six covering uniqueness of successive mnemonic calls, null-guard checks on `Wallet.from()` and `getKeyPair()`, and the `canSign()` capability of derived keypairs; plus one IMPLICIT test confirming that `Wallet.from()` with a passphrase returns a non-null object (the benchmark uses the wallet without a null-check, so the implicit behavior is tested but not explicitly asserted). In **1.5 Muxed Accounts**, the benchmark tests five behaviors covering `MuxedAccount` construction, `id` and `ed25519AccountId` round-trips, and parsing via `fromAccountId()`; the generator matched all five and expanded by four more, adding the `M`-prefix format check for `accountId`, a parsed-address equality assertion, and payment builder compatibility with M-addresses. The benchmark file also contains 11 SEP-23 test cases for low-level StrKey encoding, signed payloads, and claimable-balance IDs that come from a separate specification rather than from `sdk-usage.md`; because the generator only targets the source document, it correctly produced no tests for these, and they are excluded from the Total ACs count entirely. In **1.6 Connecting to Networks**, the benchmark tests seven behaviors — non-null checks for all three predefined SDK instances (`TESTNET`, `PUBLIC`, `FUTURENET`), their corresponding `Network.*` constant objects, and the custom-URL constructor; the generator matched all seven and added one EXPANDED test verifying that the three network passphrases are distinct, the highest ALIGNED ratio in the group (7/8, 88%) due to the fully deterministic, constant-based nature of the API.

The only subtopic that did not reach 100% benchmark alignment is **1.3 Funding Testnet Accounts**, where two of the seven ACs fall into Not Covered. The first is DIVERGENT: the benchmark asserts that `fundTestAccount()` returns `true`, but the return type is not described anywhere in `sdk-usage.md`, so the generator produced no test for this behavior — a case where the benchmark covers something the documentation does not, rather than the generator missing something it should have tested. The second is UNTESTABLE: the constraint that FriendBot only works on testnet is a network-level policy that cannot be expressed as a deterministic unit test, so neither the benchmark nor the generator attempts it. The remaining five ACs were covered normally — three ALIGNED (async invocation pattern, valid G-address input, and fresh-keypair requirement) and two EXPANDED (post-funding account existence check and a `closeTo(10000, 100)` balance assertion derived from the documentation's statement that FriendBot funds approximately 10,000 XLM). Overall, the pattern across all six subtopics is consistent: where the benchmark has tests for documented behaviors, the generator meets them; it then reliably adds EXPANDED tests for documented behaviors the benchmark does not exercise. The 35 EXPANDED tests across 70 ACs are not gaps — they are additional coverage of real documented behaviors that the human-written benchmark underrepresents.
